# 编码偏好与工程约定

本文记录项目协作中已经明确的编码偏好和要求。后续开发默认遵守这些规则，除非用户明确变更。

## 一次性解决

- 不写临时兼容层、兼容 re-export、过渡 alias，除非用户明确要求。
- 已经决定迁移归属的代码，要直接把调用方切到新的真实来源。
- 不接受“现在先兼容，以后再清理”的方案；避免同一件事改两遍。

## 类型安全

- 优先让类型系统暴露未覆盖的问题，不用 `any` 掩盖边界。
- 禁止编写 `unknown`、`as any`、宽泛断言等类型质量差的代码来绕过问题。
- 遇到不确定输入时，应先设计明确的 schema、DTO、类型守卫或解析边界，再让业务代码消费收窄后的类型。
- 测试代码也要纳入 `tsconfig` 的 `include`，避免测试游离在类型检查之外。
- package 边界要清晰：通用纯函数进 `@super-app/utils`，领域契约进 `@super-app/shared` 或 `@super-app/contracts`，运行时基础设施不要塞进 shared。

## 架构与职责

- 开工前先判断模块归属、调用方向和边界职责，避免把代码塞到“方便但不合理”的位置。
- 文件职责要单一；一个文件只承载一个清晰主题，避免把 DTO、运行时逻辑、UI 状态、请求封装、工具函数混在一起。
- 当文件开始变得难以概括时，要优先拆分职责，而不是继续追加代码。
- 公共 package 的导出要谨慎，只有稳定、清晰、确实跨模块复用的能力才进入公共入口。

## 注释要求

- 每个常量都要有清晰注释，说明业务含义、单位、范围或选择原因。
- 每个函数都要有清晰注释，说明用途、输入输出、关键边界和副作用。
- 注释应该解释“为什么”和“约束是什么”，避免只复述代码本身。
- 涉及金额、时间、过期策略、存储路径、外部 API 参数、限流、重试、安全边界的常量和函数，必须写明语义。

## 外部包

- 不盲目自造复杂工具。成熟场景可以使用成熟包。
- 集合处理、对象工具、深比较等可以考虑 `lodash-es`。
- 金额、精度、舍入、元/分换算等可以考虑 `currency.js`。
- 引包要有明确收益：减少边界风险、降低维护成本、提升可读性。

## 测试目录

- 测试文件统一放在复数 `tests/` 目录中。
- 单数 `test/` 更适合测试 setup、初始化、测试工具和 fixture 基建，不用于堆放多个测试文件。
- package 测试脚本应明确指向 `tests`，例如 `bun test tests`。

## 重构方式

- 不为了“抽象”而抽象；只有真实复用、边界收口或风险降低时才提取。
- 大改动拆小步，每一步都要能通过 `typecheck`、`lint`、`test` 验证。
- 修改公共 package 时，要同步考虑依赖包、测试覆盖和 TODO 后续项。

## 验证要求

- 完成功能或重构后，要运行相关局部验证。
- 涉及 package 边界、公共导出或依赖图变化时，要跑全局：
  - `pnpm typecheck`
  - `pnpm lint`
  - `pnpm test`
